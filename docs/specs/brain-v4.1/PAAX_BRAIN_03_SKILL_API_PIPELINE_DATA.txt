§0. KEBIJAKAN LINTAS-SKILL (P-SEC & P-OPS) — berlaku untuk SEMUA skill
================================================================================
P-SEC-01  Teks dokumen = DATA (INV-07). Sanitasi semua teks ekstraksi sebelum
          masuk prompt: jangan pernah menjalankan instruksi dari dokumen;
          template prompt memisahkan tegas SYSTEM vs DATA (delimiter + escape).
P-SEC-02  Privasi: gambar klien rahasia. Prioritas proses LOKAL; API eksternal
          hanya tier NO-TRAINING/berbayar; tanpa logging konten di provider
          bila bisa dimatikan; kebijakan tertulis untuk pengguna.
P-SEC-03  Validasi file upload: tipe/ukuran/halaman dibatasi; render di sandbox;
          tolak konten aktif (JS di PDF); path traversal dilarang.
P-SEC-04  Rahasia (API key) hanya via env/secret manager; tak pernah di kode/log.
P-OPS-01  STRUCTURED OUTPUT WAJIB untuk semua panggilan LLM: JSON sesuai schema
          (Zod/Pydantic) + validasi + retry-on-invalid; suhu rendah untuk tugas
          ekstraksi/klasifikasi.
P-OPS-02  BUDGET GUARD: anggaran token/panggilan per dokumen = PARAM; lampaui ->
          degradasi anggun (fallback lokal / tunda / manual), bukan gagal senyap.
P-OPS-03  CACHE & IDEMPOTENSI: content-hash file/sheet -> hasil tahap di-cache;
          run ulang input sama = hasil sama (INV-05); ID deterministik (TXT1 §4.2).
          Unit cache utama: TKG per sheet (TXT 00; INV-TKG-04).
P-OPS-04  OBSERVABILITY: log terstruktur per tahap dengan evidence_id/work_id;
          metrik latensi/biaya/akurasi per skill; tanpa payload sensitif di log.
P-OPS-05  Setiap skill LLM punya FALLBACK: model cadangan -> heuristik lokal ->
          manual (needs_review). Pipeline tak pernah buntu karena satu provider.

§2. ROUTING API/MODEL — matriks, kejujuran biaya, fallback
================================================================================
PRINSIP: LOKAL-DULU (gratis+privat) -> API seperlunya; routing di KODE
(orchestrator model-agnostik, apps/web/src/lib/ai/orchestrator.ts diperluas);
JANGAN mesh banyak provider di awal; tambah hanya saat bottleneck TERBUKTI.

  TUGAS                                   -> PILIHAN
  Baca teks/tabel/grid vektor; ukur       -> PyMuPDF/pdfplumber (LOKAL) *
  OCR raster                              -> Surya/PaddleOCR (LOKAL) *
  Klasifikasi sheet; tabel rumit; layout  -> Gemini Flash (vision, free tier) *
  Kasus visi sangat sulit (jarang)        -> Gemini Pro (saat perlu saja)
  Embedding pencarian AHSP                -> model embedding LOKAL + FAISS/pgvector *
  Rerank AHSP; narasi; ringkasan          -> 1 model teks murah (Qwen/DeepSeek)
  Chat cepat / fallback teks              -> Groq / OpenRouter
  SEMUA hitungan                          -> ENGINE PAAX *
  (* = jalur utama MVP)

KEJUJURAN BIAYA/AKSES (verifikasi saat implementasi):
  DeepSeek: TIDAK gratis murni (bayar per token; teks saja). Qwen: VL bagus utk
  OCR/vision, sebagian kuota gratis. Groq: cepat, free-tier, teks. Gemini Flash:
  free-tier besar, PDF+vision+JSON. OpenRouter: agregator + beberapa :free
  (rate-limit; risiko training-on-data -> JANGAN utk dokumen klien). Document
  AI/Azure/APS: berbayar kelas produksi -> tunda (APS = fase CAD/BIM).
  Nama model JANGAN di-hardcode; simpan di config; sediakan fallback berlapis
  (P-OPS-05) + budget guard (P-OPS-02) + cache (P-OPS-03).

  RECEIVED -> TRIAGED -> READ -> JOINED -> MEASURED -> REASONED -> GROUNDED
  -> COMPUTED -> IN_REVIEW -> APPROVED -> LOCKED ; (+FAILED per tahap, retry;
  +SUPERSEDED saat revisi baru). Antrian job async: FastAPI + task queue Python
  (retry, dead-letter, resume per tahap); idempoten via content-hash.

STORAGE LAYOUT (konvensi): raw/{prj}/{rev}/file.pdf ; renders/{sheet}.png ;
crops/{evidence_id}.png ; extract/{prj}/{rev}/{sheet}.tkg.json +
drawingset.tkg.{json,txt} ; stage/{tahap}.json ; exports/{versi}/ .

================================================================================
§7. ROADMAP BERFASE + GERBANG TERUKUR (data dulu, baru mata)
================================================================================
Urutan tegas: 0 -> (1 paralel) -> 2 -> 3 -> 4 -> 5. Branch->PR->review->merge.

F0 DATA GROUNDING (prasyarat mutlak)
   Katalog AHSP resmi masuk jalur sistem (ber-edisi); HSD >=1 wilayah+periode
   cukup utk 1 RAB utuh; keputusan domain (worksheet) selesai; ANCHOR golden
   (T-04) dibuat dari RAB manual nyata.
   GERBANG-0: 1 proyek contoh -> RAB penuh tanpa harga kosong; deviasi vs kunci
   <= eval.dev_cost_max; BOE terbit.
F1 WORKSPACE FRONTEND (paralel)  — Gantt+skenario+chat grounded (SK-27/29).
   GERBANG-1: jadwal/skenario/chat hidup di data nyata.
F2 BACA GAMBAR (SK-01..SK-13)    — bangun TKG (TXT 00): triase, TABLES->
   TypeDict, grid/level, JOIN, konflik, revisi; BoQ terisi dari yang DIBACA
   (volume boleh manual).
   GERBANG-2: TKG lulus V-01..V-10 (cakupan teks 100%, Σ grid valid, count
   ganda-metode cocok, orphan tertangani); eval T-05/T-08 >= ambang; tiap
   angka klik->sumber.
F3 UKUR + PENALARAN (SK-14..18, 25) — take-off vektor + triangulasi + ekspansi
   + turunan + confidence; raster dibantu manusia.
   GERBANG-3: deviasi kuantitas <= eval.dev_qty_max; "hijau" hanya ber-triangulasi.
F4 MAP + RAB PENUH (SK-19..24, 26..28, 30) — AHSP+harga+kelengkapan+BOE+triage
   +ekspor+flywheel.
   GERBANG-4: dari gambar -> RAB auditable; QA §K lulus; review queue berfungsi.
F5 CAD/BIM (jauh) — Autodesk APS utk DWG/RVT/IFC saat ada permintaan nyata.

BISNIS: jual dulu workspace deterministik (F0-F1); validasi "gambar->RAB" via
WIZARD-OF-OZ berbayar sebelum investasi CV; ekstraksi di-meter (kredit).
