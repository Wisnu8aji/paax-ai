§2. POSISI PAAX, HARGA BERLAPIS, MODE PRODUK
================================================================================
"Harga" berlapis; agent wajib sadar level keluarannya:
  L1 Estimasi kasar (owner)            -> mode ESTIMATE (boleh)
  L2 RAB / Engineer Estimate (QS)      -> LEVEL UTAMA PAAX (mode RAB)
  L3 HPS (PPK)                         -> BUKAN wewenang PAAX (alat bantu saja)
  L4 Harga penawaran (kontraktor)      -> mode TENDER (bantu susun)
  L5 Nilai kontrak                     -> di luar lingkup
  L6 RAP (internal kontraktor)         -> lanjutan (mode MONITOR terkait)

RULE-POS-01  Keluaran default = RAB/Engineer Estimate. Jangan pernah melabeli
             keluaran sebagai HPS resmi, harga kontrak, atau hasil desain
             struktur. Sisipkan disclaimer decision-support pada keluaran.
RULE-POS-02  Untuk proyek pemerintah: HPS/penetapan pemenang = wewenang PPK/
             Pokja; PAAX hanya membantu menyusun/memeriksa.
RULE-POS-03  Titik masuk rantai proyek: GAMBAR/DED (+RKS/BQ) -> take-off -> BoQ
             -> RAB -> jadwal awal (+ dukungan tender & monitor).
RULE-POS-04  Regulasi acuan: Permen PUPR No. 8/2023 (AHSP) + SE DJBK; SMKK
             mengacu pedoman SMKK (Permen PUPR No. 10/2021). Struktur keluaran
             mengikuti kerangka ini.
RULE-MODE-EST  Mode ESTIMATE: biaya parametrik dari indeks biaya per m2
             (katalog indeks: tipe bangunan x kelas x wilayah x periode; F-M01)
             + rentang ketidakpastian + daftar data yang kurang. TIDAK memakai
             indeks karangan; bila katalog indeks kosong -> needs_review.
RULE-MODE-RAB  Mode RAB (MVP): pipeline penuh TXT ini (gambar -> BoQ -> RAB).
RULE-MODE-TDR  Mode TENDER: bila BQ pemberi kerja tersedia, JANGAN mengulang
             take-off kecuali diminta; isi harga satuan via engine (AHSP+HSD),
             ringkas lingkup/risiko/pertanyaan aanwijzing. Perbandingan ke HPS
             hanya bila HPS diberikan pengguna.
RULE-MODE-MON  Mode MONITOR: progres (opname) vs rencana -> kurva-S aktual,
             deviasi, EVM (F-I10..): murni dari data; alert via ambang PARAM.

§3. MODEL MENTAL GAMBAR + STRATEGI MEMBACA + PRESEDENSI + LOCALE
================================================================================
POLA INTI (JOIN): kuantitas = gabungan
  SCHEDULE/TABEL (definisi tipe: dimensi, tulangan, mutu, tebal, spec)
  x DENAH (instance: posisi + jumlah per tipe)
  x GRID (panjang/bentang pasti antar-as)
  x LEVEL/PEIL (tinggi dari selisih elevasi tertulis)
  x DETAIL/POTONGAN (dimensi vertikal, lapisan, penampang)
  x NOTES/RKS (mutu, campuran, merk-setara, metode).

RULE-READ-01  URUTAN BACA (strategi, bukan sekadar urutan halaman):
  (1) cover + daftar gambar + title-block (identitas, disiplin, skala, revisi)
  (2) legenda & SEMUA schedule/tabel -> bangun TypeDict (kamus tipe) dulu
  (3) denah per disiplin -> ElementInstance (posisi/jumlah) via JOIN ke TypeDict
  (4) potongan/detail -> dimensi vertikal, kedalaman, lapisan
  (5) notes/RKS/spesifikasi -> mutu/material/metode
  (6) MEP & arsitektur overlay -> titik, jalur, finishing
  (7) rekonsiliasi lintas-sheet (grid konsisten? level konsisten? tipe tanpa
      definisi? definisi tanpa instance?) -> warning.

RULE-READ-02  PRESEDENSI SAAT SUMBER KONFLIK (default; dapat dikonfigurasi via
  PARAM.precedence_order; SETIAP resolusi konflik -> Warning + Assumption; bila
  dampak biaya besar -> needs_review):
  (1) REVISI lebih baru > lama (dan addendum/BA-aanwijzing > gambar tender)
  (2) DETAIL berskala lebih besar > denah berskala kecil (untuk dimensi elemen)
  (3) SCHEDULE/TABEL = otoritas properti TIPE; DENAH = otoritas JUMLAH & POSISI
  (4) DIMENSI TERTULIS > hasil ukur skala/vektor (bila keduanya ada dan beda)
  (5) RKS/spesifikasi tertulis = otoritas MATERIAL/MUTU bila gambar diam
  (6) Bila tetap ambigu -> needs_review; JANGAN memilih diam-diam.

RULE-READ-03  LOCALE ANGKA: dokumen Indonesia lazim koma-desimal dan titik-ribuan.
  Semua parsing angka lewat normalisasi locale (F-A00); locale DIDETEKSI per
  dokumen dari bukti internal (default id-ID; TXT 00 §2.6) dan dicatat sebagai
  Assumption;
  raw_reading disimpan apa adanya. Koreksi OCR domain (0/O, 1/I/l, 5/S, mm vs m)
  hanya via kamus koreksi + ditandai (SK-10), tidak diam-diam.

RULE-READ-04  MULTI-DISIPLIN: register elemen dipisah per disiplin (STR/ARS/MEP)
  lalu direkonsiliasi; konflik lintas disiplin (mis. dinding ARS menembus balok
  STR, void tidak sinkron) -> W-XDIS + needs_review, bukan dipaksakan.

RULE-READ-05  VECTOR-FIRST: PDF vektor (ekspor CAD) = jalur utama — teks & garis
  berkoordinat -> baca & ukur deterministik, lokal, gratis, privat. Raster/scan
  = jalur bantu (OCR lokal + kalibrasi/poligon manual). Model vision TIDAK
  dipakai untuk mengukur (INV-06), hanya memahami layout/tabel/klasifikasi.

RULE-READ-06  OPERASIONALISASI & ARTEFAK: seluruh mekanik membaca (L1–L5)
  didefinisikan penuh di TXT 00 dan MENGHASILKAN TKG. Lapisan penalaran (L6 ke
  atas) HANYA membaca TKG — tidak pernah membaca berkas gambar langsung.
  Dalam semua aturan set ini, rujukan ke SCHEDULE/DENAH/GRID/LEVEL/DETAIL/
  NOTES berarti blok TABLES/ELEMENTS/GRID/LEVELS/DETAILS/NOTES di dalam TKG.

§4. MODEL ENTITAS, KONTRAK EVIDENCE, LIFECYCLE
================================================================================
§4.1 ENTITAS INTI (skema detail di TXT 3 §4; padankan Zod<->Pydantic):
  ProjectContext   : identitas proyek, mode, wilayah+periode harga, edisi AHSP,
                     tipe bangunan, precedence, parameter aktif (snapshot).
  DrawingSet       : kumpulan berkas gambar per REVISI (rev_id, tanggal, status).
  Sheet            : per lembar (sheet_id, disiplin, jenis, skala, vektor?/raster).
  ElementType      : entri TypeDict (type_id, kelas elemen, dimensi, tulangan,
                     mutu, tebal, spec) — sumber: schedule/detail.
  ElementInstance  : kemunculan tipe (element_id, type_id, level, posisi/grid,
                     count, geometri dasar refs).
  WorkItem         : item pekerjaan BoQ (work_id, divisi WBS, uraian kanonik,
                     satuan, elemen_ref[], asal {extracted|expanded|implied|
                     derived}, rule_id?, ahsp_ref?, price_ref?, volume_ref?).
  Evidence         : bukti (di bawah).
  Assumption       : entri ledger asumsi/parameter/konvensi terpakai (§6.11).
  Warning          : kode W-/E- + pesan + tautan objek.
  ReviewTask       : unit kerja review manusia (target, alasan, prioritas).
  Correction       : koreksi manusia (flywheel; field, lama->baru, alasan).

§4.2 SKEMA ID DETERMINISTIK (traceability):
  element_id = {PRJ}.{REV}.{DISC}.{LEVEL}.{TYPE}.{SEQ}
  work_id    = {PRJ}.{DIV}.{KODE-URAIAN-KANONIK}.{SEQ}
  evidence_id= {PRJ}.{REV}.{SHEET}.{KIND}.{SEQ}
  ID stabil antar-run untuk input identik (idempotensi; TXT 3 §3).

§4.3 KONTRAK EVIDENCE ("teks yang bisa dijalankan"):
  Evidence {
    id, project_id, revision_id, discipline
    kind    : enum {
      // BACA (tergambar):
      dimension_text, room_label, schedule_row, type_tag, grid_span, level_mark,
      title_block, spec_note, scale, legend_entry,
      // UKUR (geometri):
      measured_length, measured_area, measured_volume, symbol_count,
      // PENALARAN (tersirat/turunan):
      expanded_work, implied_work, derived_quantity, dependency_link, assumption_ref
    }
    value, unit
    element_ref? , work_ref?
    source  : { sheet_id?, page?, locator (bbox|vector_ref)?, raw_reading? }
    method  : enum { read_from_text, read_from_table, read_from_grid,
                     read_from_level, vector_length, vector_polygon,
                     symbol_detect, manual_two_point, manual_input,
                     reasoned_rule, derived_formula }
    engine_or_model : enum {pymupdf, ocr_local, gemini, qwen, deepseek, human, engine}
    rule_id?        // WAJIB bila method=reasoned_rule/derived_formula
    rationale?      // alasan singkat penalaran (audit)
    confidence      // DIHITUNG (F-J*), bukan dikarang
    corroborated_by[] , conflicts_with[]
    review_state    : enum lifecycle (§4.4)
    superseded_by?  // terisi bila revisi baru menggantikan
    warnings[]
  }

§4.4 LIFECYCLE (state machine Evidence/WorkItem):
  EXTRACTED -> CORROBORATED (lolos triangulasi) -> [NEEDS_REVIEW bila §6.10]
  -> APPROVED (manusia) -> LOCKED (masuk RAB final).
  Transisi tambahan: * -> SUPERSEDED (revisi baru; §6.9) -> kembali NEEDS_REVIEW
  pada objek pengganti. Semua transisi tercatat (siapa/kapan/kenapa).
  INV-11  Hanya Evidence ber-state APPROVED/LOCKED yang boleh menjadi dasar
          RAB final; NEEDS_REVIEW boleh tampil di draft dengan tanda jelas.

§5. ARSITEKTUR KOGNITIF — 9 LAPIS + STATE DOKUMEN
================================================================================
Pipeline (tiap panah = Evidence tervalidasi):
  Upload -> L1 TRIASE (split, vektor/raster, grid, skala)
         -> L2 LAYOUT (title-block, tabel, denah, notes)
         -> L3 BACA (fakta tertulis; TypeDict; grid; level)
         -> L4 SEMANTIK (klasifikasi; JOIN instance<->tipe; resolusi konflik;
                         diff revisi)
         -> L5 UKUR (geometri vektor deterministik; raster dibantu manusia)
         -> [ARTEFAK: TKG — hasil L1–L5; kontrak TXT 00 §6; satu-satunya
             pintu fakta gambar bagi lapisan berikutnya]
         -> L6 PENALARAN (§6: ekspansi, tersirat, turunan, AHSP, harga,
                         dependensi, kelengkapan, triage, asumsi)
         -> L7 GROUNDING (Evidence + confidence + triangulasi)
         -> L8 ENGINE (take-off -> HSP -> RAB -> bobot -> jadwal/CPM/EVM)
         -> L9 REVIEW (approve/edit/reject; Correction -> flywheel; LOCK).

STATE DOKUMEN (per DrawingSet; detail TXT 3 §3): RECEIVED -> TRIAGED -> READ ->
JOINED -> MEASURED -> REASONED -> GROUNDED -> COMPUTED -> IN_REVIEW -> APPROVED
-> LOCKED (+ FAILED, SUPERSEDED). Idempoten via content-hash; hasil tahap
di-cache; tahap gagal bisa diulang tanpa merusak tahap lain.

Pemetaan repo: L1–L7 = services/document-intelligence (Lapis 2A);
L8 = services/core-engine (Lapis 2B; SEMUA rumus hidup di sini);
routing model = orchestrator (Lapis 1); L9 UI = apps/web (Lapis 0);
katalog AHSP/HSD/indeks + object storage + vector index = Lapis 3.


================================================================================
§6. MESIN PENALARAN (REASONING CORE)
================================================================================
Prinsip: RAB PROFESIONAL = TERGAMBAR + TERSIRAT. Banyak isi RAB tidak tergambar
dan wajib ditalar dengan logika konstruksi — secara auditable.

RULE-RAB-01  Dua sumber WorkItem, keduanya wajib ditelusuri:
  (A) TERGAMBAR: hasil L3–L5 (elemen & kuantitas terbaca/terukur).
  (B) TERSIRAT : hasil L6 (ekspansi §6.2, implied §6.3, turunan §6.4).
RULE-RAB-02  Setiap item TERSIRAT wajib Evidence method=reasoned_rule/
  derived_formula + rule_id + rationale (bisa dikonfirmasi/ditolak manusia).
RULE-RAB-03  Kuantitas item tersirat berasal dari: rumus (TXT 2) ATAU parameter
  bernama ATAU needs_review manusia — TIDAK PERNAH dikarang.

--------------------------------------------------------------------------------
§6.2 EKSPANSI ELEMEN -> PEKERJAAN (pengetahuan QS, generic)
--------------------------------------------------------------------------------
RULE-EXP-BETON  Setiap elemen beton cor-di-tempat (footplat/pilecap, sloof,
  kolom, kolom praktis, balok, ring balok, latei/lintel, pelat/dak, talang
  beton, kanopi, topi, tangga+bordes, dinding beton, parapet) mekar menjadi:
    (1) BETON [m3]  — kelas/mutu sesuai schedule (item AHSP per kelas)
    (2) PEMBESIAN [kg] — konfigurasi dari schedule (utama+sengkang+pinggang+susut)
    (3) BEKISTING [m2] — bidang kontak sesuai tipe (rumus TXT 2 §C)
  Catatan bidang: dasar footplat & pelat-di-atas-tanah tanpa acuan bawah; sloof
  di atas lantai kerja umumnya 2 sisi; kolom menempel dinding -> sisi tempel
  dikurangi (F-C08). Elemen ambigu -> konvensi konservatif + needs_review.
RULE-EXP-BETON-MUTU  Kelas beton berbeda = item AHSP berbeda; beton struktur vs
  non-struktur (lantai kerja/rabat) TIDAK dicampur. Padanan penamaan K<->fc'
  hanya via F-B10 (konvensi), nilai desain tetap dari gambar.
RULE-EXP-PONDASI  Pondasi menuntut pendamping: LANTAI KERJA (F-B09), PONDASI
  MENERUS batu belah bila ada (F-G01), plus rantai TANAH (§6.3/6.4). Kedalaman
  dari detail/level, bukan asumsi.
RULE-EXP-DINDING  Bidang dinding pasangan -> rantai: PASANGAN [m2] (jenis+
  campuran per spek) -> PLESTERAN [m2] (sisi sesuai kondisi) -> ACIAN [m2] ->
  SPONNINGAN/tali air [m'] (bila spek) -> CAT [m2]. Di atas bukaan -> LATEI
  (panjang = lebar bukaan + 2 x PARAM.L_bearing). Penambah kekakuan: KOLOM
  PRAKTIS & RING praktis bila panjang segmen > PARAM.L_maks_praktis ATAU luas
  panel > PARAM.A_maks_praktis (keduanya keputusan praktik -> Assumption).
RULE-EXP-LANTAI  Lantai -> lapisan: URUGAN pasir/sirtu (di atas tanah) -> RABAT/
  screed bila spek -> PENUTUP lantai [m2] -> PLIN [m'] (dikurangi lebar pintu).
  Area basah: WATERPROOFING (bidang + upstand; F-G10) + keramik dinding
  (tinggi pasang dari spek/PARAM.h_pasang_keramik).
RULE-EXP-ATAP  Atap -> RANGKA (baja profil per kg dari tabel profil x panjang
  member; baja ringan per m2 bidang miring) -> PENUTUP [m2 miring] -> NOK [m']
  -> LISPLANK [m'] -> TALANG & PIPA AIR HUJAN [m']. Elemen ikat (gording,
  trekstang, ikatan angin, plat buhul, baut) dibaca dari denah atap/detail.
RULE-EXP-KUSEN  Kusen/pintu/jendela mengikuti SCHEDULE sebagai kamus tipe:
  kusen [m'/unit] + daun + kaca [m2] + aksesoris [bh] + jalusi/boven bila ada.
  Jumlah unit = count denah x schedule (triangulasi RULE-CONF-02).
RULE-EXP-MEP  MEP -> instalasi per TITIK/METER + fixture/armatur [bh/unit] +
  panel/proteksi. Jalur pipa dari denah plumbing; bila denah tak ada ->
  needs_review / parameter per fixture (Assumption).
RULE-EXP-FINISH  Finishing pelengkap: PLAFON (rangka+penutup+list), CAT
  (dinding/plafon/besi), WATERPROOFING, RAILING/HANDRAIL untuk tangga/void/ram
  [m'], fasad khusus (batu alam/GRC/ACP/partisi) sesuai gambar arsitektur.
RULE-EXP-OWN  (penegasan INV-09/10) Ekspansi lewat register elemen unik; tiap
  kuantitas dimiliki satu WorkItem; deduksi bukaan didefinisikan sekali.

--------------------------------------------------------------------------------
§6.3 PEKERJAAN TERSIRAT / NON-GAMBAR (implied works)
--------------------------------------------------------------------------------
RULE-IMP-SMKK  SMKK/K3 wajib (khususnya pemerintah): dokumen RKK, sosialisasi/
  pelatihan, APD/APK, asuransi, personel K3, P3K, rambu, pengendalian risiko.
  Divisi tersendiri; komponen mengikuti pedoman SMKK; besaran = parameter/
  keputusan -> needs_review bila belum ditetapkan.
RULE-IMP-PERSIAPAN  Persiapan: pembersihan lahan & removal, pengukuran +
  BOUWPLANK (kuantitas proporsional keliling bangunan; F-K01), direksi keet/
  gudang, pagar proyek, papan nama, MOBILISASI/DEMOBILISASI, air & listrik kerja.
RULE-IMP-METODE  Pekerjaan metode/temporary berkonsekuensi biaya, dipicu kondisi:
  SEWA POMPA beton (volume per pengecoran > PARAM.V_pompa_min ATAU tinggi >
  PARAM.H_pompa_min), PERANCAH (hanya bila AHSP bekisting TIDAK termasuk
  perancah — cek uraian; RULE-AHSP-05), dewatering (muka air tinggi), pengaman
  galian (kedalaman > ambang), jalan kerja. Semua trigger = parameter + Assumption.
RULE-IMP-TANAH  Tanah turunan: galian, urugan kembali, urugan pasir/sirtu,
  pemadatan, BUANGAN tanah sisa + angkutan (F-F07) — jangan lupa buangan.
RULE-IMP-UJI   Pengujian & commissioning: uji beton (slump, benda uji) per
  volume beton (frekuensi = parameter/spek), tes instalasi MEP, commissioning.
RULE-IMP-HALAMAN  Site: pemadatan, paving/perkerasan, saluran + tutup, taman.
RULE-IMP-AKHIR  Akhir: pembersihan akhir, as-built & dokumentasi, administrasi
  serah terima (PHO/FHO) — item administratif ditandai non-fisik.
INV-12  Divisi tersirat relevan yang datanya kurang -> item needs_review +
  W-ASM (bukan dihilangkan diam-diam; RAB tampak lengkap DAN jujur).

§6.4 KUANTITAS TURUNAN (derived)
--------------------------------------------------------------------------------
Diturunkan deterministik dari geometri + parameter (rumus TXT 2):
  galian (F-F01/02), urugan kembali (F-F03), pasir/sirtu (F-F04/05), BUANGAN +
  angkut (F-F07), lantai kerja (F-B09), screed (F-E07), plesteran/acian
  (F-E02/03), cat (F-E05), bekisting (F-C*), besi (F-D*), waterproofing+upstand
  (F-G10), bouwplank (F-K01). LLM hanya memilih rumus & kondisi; engine
  mengeksekusi (INV-01).

--------------------------------------------------------------------------------
§6.5 PEMILIHAN AHSP (pencarian, bukan generasi)
--------------------------------------------------------------------------------
RULE-AHSP-01  Tiap WorkItem dipetakan via SEMANTIC SEARCH ke katalog AHSP resmi
  (index embedding) -> top-k -> LLM rerank + alasan -> manusia konfirmasi.
RULE-AHSP-02  Fitur yang WAJIB cocok: jenis pekerjaan; MUTU/kelas; CAMPURAN/
  rasio; dimensi/spek yang mengubah analisa; METODE; dan SATUAN.
RULE-AHSP-03  SATUAN AHSP harus sama dengan satuan WorkItem; bila beda ->
  konversi eksplisit (rumus/parameter, tercatat sebagai Assumption) atau tolak
  kandidat. Dilarang mengalikan diam-diam.
RULE-AHSP-04  Dilarang mengarang kode/koefisien. Tak ada padanan meyakinkan ->
  needs_review + kandidat terdekat + W-AHSP.
RULE-AHSP-05  ANTI DOUBLE-COUNT KANDUNGAN: baca uraian/komponen AHSP — bila item
  sudah "termasuk" sub-pekerjaan (mis. bekisting termasuk perancah; beton
  termasuk bekisting+besi pada analisa gabungan), JANGAN membuat item terpisah
  untuk sub tersebut. Sebaliknya bila analisa polos, sub-pekerjaan berdiri
  sendiri. Keputusan tercatat (Assumption).
RULE-AHSP-06  Edisi/versi katalog AHSP eksplisit di ProjectContext; jangan
  mencampur edisi dalam satu RAB.

--------------------------------------------------------------------------------
§6.6 PEMILIHAN HARGA DASAR (HSD)
--------------------------------------------------------------------------------
RULE-HRG-01  Harga dari katalog HSD per WILAYAH + PERIODE (Lapis 3); dicocokkan
  per resource (kode/nama) -> engine mengalikan koef x harga.
RULE-HRG-02  Prioritas sumber: SHSD/harga resmi > survey terverifikasi > default
  parameter (ditandai Assumption). Harga kosong -> missing + W-PRC; RAB final
  tidak boleh mengandung harga nol tersembunyi; cakupan harga dilaporkan.
RULE-HRG-03  Wilayah+periode tampil pada keluaran (audit). Dilarang mencampur
  wilayah/periode tanpa keputusan eksplisit (eskalasi = keputusan owner).
RULE-HRG-04  HSP = (Upah + Bahan + Alat) x (1 + PARAM.OP); baris RAB = Volume x
  HSP; PPN = PARAM.tarif_PPN (regulasi berlaku). Rumus penuh TXT 2 §H.

--------------------------------------------------------------------------------
§6.7 KELENGKAPAN, KONSISTENSI, TAKSONOMI WARNING
--------------------------------------------------------------------------------
RULE-CHK-01  Cek kelengkapan gambar vs kebutuhan RAB (per WBS §9): laporkan ADA
  vs BELUM (mis. denah arsitektur, detail pondasi, schedule tulangan, denah MEP,
  spesifikasi finishing).
RULE-CHK-02  Konsistensi lintas-sheet & gambar-vs-BoQ: grid/level konsisten;
  tipe tanpa definisi; definisi tanpa instance; jumlah denah vs BoQ; duplikasi.
RULE-CHK-03  Warning menempel pada objek (Evidence/WorkItem), bukan catatan lepas.
TAKSONOMI KODE (contoh kanonik; daftar hidup di kode):
  W-SCL-01 skala tak terbaca            | W-TYP-01 tipe dipakai tanpa definisi
  W-DEF-01 definisi tanpa instance      | W-CNT-01 jumlah simbol meragukan
  W-SPE-01 spesifikasi material kosong  | W-PRC-01 harga resource kosong
  W-ASM-01 parameter default dipakai    | W-REV-01 revisi lebih baru terdeteksi
  W-XDIS-01 konflik lintas disiplin     | W-AHSP-01 mapping AHSP meragukan
  E-UNT-01 satuan tak cocok (kritis)    | E-DUP-01 duplikasi elemen (kritis)
  E-GRID-01 grid tak konsisten (kritis) | E-REVMIX-01 campur revisi (kritis)

--------------------------------------------------------------------------------
§6.8 PENALARAN DEPENDENSI KONSTRUKSI (untuk jadwal)
--------------------------------------------------------------------------------
RULE-DEP-01  AI mengusulkan graf dependensi antar-WorkItem dari logika
  konstruksi; default Finish-to-Start; overlap/lag = parameter; manusia
  mengonfirmasi. Engine menghitung CPM (TXT 2 §I).
RULE-DEP-02  Rantai default (disesuaikan konteks): persiapan -> tanah ->
  (lantai kerja -> pondasi -> sloof) -> per lantai n: (kolom lt-n -> balok+pelat
  lt-n) -> ring -> rangka atap -> penutup atap; dinding lt-n boleh mulai setelah
  struktur lt-n; MEP rough-in setelah dinding; plafon setelah rough-in atas;
  lantai finishing setelah basah selesai; kusen sebelum finishing cat akhir;
  sanitair fixture akhir; halaman paralel akhir; pembersihan/commissioning
  terakhir. Semua tautan = Evidence(kind=dependency_link, rule_id).
RULE-DEP-03  Jangan menjadwalkan item needs_review sebagai kritis tanpa tanda;
  jalur kritis yang melewati item belum-approved diberi warning.

--------------------------------------------------------------------------------
§6.9 REVISI & DELTA (gambar berubah — kenyataan lapangan)
--------------------------------------------------------------------------------
RULE-REV-01  Setiap DrawingSet punya rev_id; ekstraksi terikat revisi. Dilarang
  menjumlah kuantitas lintas revisi (E-REVMIX).
RULE-REV-02  Revisi baru -> jalankan DIFF (SK-13): elemen/tipe berubah ->
  Evidence lama SUPERSEDED; WorkItem terdampak kembali NEEDS_REVIEW; laporan
  delta (tambah/kurang/ubah) untuk manusia (dasar addendum/CCO).
RULE-REV-03  RAB LOCKED tidak diedit; revisi menghasilkan RAB versi baru dengan
  jejak ke versi sebelumnya.

--------------------------------------------------------------------------------
§6.10 TRIAGE REVIEW BERBASIS RISIKO
--------------------------------------------------------------------------------
RULE-TRI-01  WAJIB review manusia bila salah satu:
  (a) item termasuk kelompok biaya kumulatif teratas >= PARAM.p_pareto;
  (b) confidence < PARAM.ambang_conf;
  (c) kuantitas kritis tanpa korroborasi (triangulasi gagal/absen);
  (d) item implied berdampak biaya besar; (e) hasil resolusi konflik presedensi.
  ReviewTask dibuat otomatis, diurut prioritas (dampak x ketidakpastian).

--------------------------------------------------------------------------------
§6.11 DAFTAR ASUMSI / BASIS OF ESTIMATE (BOE) — keluaran wajib
--------------------------------------------------------------------------------
RULE-BOE-01  Setiap RAB/estimasi disertai BOE berisi: (i) daftar Assumption
  (parameter default, konvensi, resolusi presedensi, konversi satuan);
  (ii) sumber data (edisi AHSP, wilayah+periode HSD, cakupan harga);
  (iii) daftar kekurangan data (missing) + dampaknya; (iv) ringkasan warning.
RULE-BOE-02  BOE dihasilkan engine dari ledger (bukan dikarang LLM) dan ikut
  ter-lock bersama RAB.


================================================================================
§7. CONFIDENCE (DIHITUNG) + TRIANGULASI
================================================================================
RULE-CONF-01  Confidence dihitung fungsi deterministik (F-J*), dari: keandalan
  metode (grid/tabel/vektor > OCR > visual), kualitas baca, korroborasi.
  Sinyal confidence LLM hanya salah satu input, tak pernah final (AP-04).
RULE-CONF-02  Triangulasi: kuantitas "HIJAU" hanya bila >=2 sumber independen
  setuju dalam toleransi (F-J04) — mis. panjang grid vs dimensi tertulis; count
  simbol vs jumlah titik grid vs schedule; luas poligon vs perkalian dimensi.
RULE-CONF-03  Gagal/absen korroborasi pada kuantitas kritis -> NEEDS_REVIEW.


================================================================================
§8. HUMAN-IN-THE-LOOP & GERBANG
================================================================================
RULE-HITL-01  UI review: gambar + overlay area terhitung + nilai + sumber +
  confidence + warning + approve/edit/reject; setiap angka bisa diklik ->
  Evidence (termasuk rule_id+rationale untuk item tersirat).
RULE-HITL-02  Edit manusia -> Correction (flywheel) + state APPROVED; LOCK saat
  finalisasi; revisi membuka siklus baru (§6.9).
RULE-HITL-03  Flywheel dipakai untuk: perbaikan prompt/threshold (pendek),
  peringkat mapping AHSP (menengah), kalibrasi/fine-tune (panjang).
RULE-HITL-04  Gerbang kode: branch->PR->review->merge; no auto-merge (CLAUDE.md).
